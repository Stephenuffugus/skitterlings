Same origin + same (empty) uid, but two different partitions = two collections.

PROPOSED SAVE-BRIDGE CONTRACT (tell us if you'd rather shape it differently):

1. IDENTITY. We will pass the LW player's uid two ways (use whichever is easier):
   - launch param: ...?embed=1&origin=https://lucidwinds.com&uid=<LW_UID>
   - and/or a postMessage after your 'ready':
     { target:"skitterlings", type:"configure", data:{ uid:"<LW_UID>" } }
   For a signed-OUT player there is no uid; keep using localStorage (per-browser)
   in that case — that's fine, sign-in is what unlocks cross-device.

2. LOAD (host -> game). After we receive your existing `ready` event, we post the
   player's saved blob (or null if none):
     { target:"skitterlings", type:"loadState", data:{ <your save blob> } }
   Your side: on receiving this, MERGE it into the local save (prefer the
   incoming server blob as authoritative for owned skins/worlds/best; union is
   safest). If data is null, keep whatever is local.

3. SAVE (game -> host). On every meaningful change (skin unlocked, world cleared,
   best score), emit your full save blob:
     { source:"skitterlings", type:"state", data:{ <your save blob> } }
   Debounced is fine (e.g. coalesce to once every ~2-5s). We persist the latest
   blob per-uid. Keep `coins` OUT of this (coins stay internal as agreed) OR
   include them and we'll ignore them — your call, just be consistent.

4. We do the rest: validate origin (already in STUDIO_ORIGINS), persist the blob
   to the vault (debounced), and replay it on the next load on any device the
   same uid signs into.

That's it — one event you emit (`state`) and one you accept (`loadState`), plus
reading the uid. The collection becomes authoritative server-side and identical
everywhere the same signed-in player plays.
